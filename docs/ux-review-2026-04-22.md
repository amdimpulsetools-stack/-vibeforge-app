# UX Review — 2026-04-22

Senior UI/UX audit of the VibeForge medical SaaS (Next.js 15 + Tailwind + shadcn/ui). Roles: doctors, admins, receptionists (desktop) + patients (mobile portal). LATAM, Spanish.

---

## Executive summary

1. **Two parallel design systems coexist and collide.** The patient portal (`/portal`) runs an Apple-flavored system (iOS greys `#F2F2F7`, zinc palette, custom hex pills, `rounded-3xl`). The dashboard (`/(dashboard)`) runs a shadcn-on-tokens system (`bg-card`, `border-border`, `rounded-xl`, emerald accent). The public booking page (`/book`) is a light third variant. Tokens are not shared; fixes in one don't propagate.
2. **Dashboard modals violate WCAG out of the box.** Zero `role="dialog"` / `aria-modal` / focus traps / ESC handlers on `appointment-form-modal.tsx`, `appointment-sidebar.tsx` actions, `patient-drawer.tsx` modals. The only compliant modal in the codebase is inside the patient portal.
3. **Destructive actions use native `confirm()` / `alert()`.** Patient portal cancel flow uses `alert()`; scheduler delete uses `window.confirm()`; clinical-note sign uses `confirm()`. No undoable patterns, no typed confirmation for irreversible sign.
4. **Public booking (/book/[slug]) drops users into a wall of steps with no save state and poor input hygiene.** No field-level validation messaging, no phone/DNI format mask, "Error al crear la cita" is the only failure UI. Highest-risk revenue flow has the weakest safety net.
5. **Text contrast and touch targets skate under AA on several high-traffic surfaces.** `text-[10px]`/`text-[9px]` appears 24+ times in scheduler sidebar; 32px hit targets on drawer close buttons; secondary-label greys (`text-zinc-500` on `#F2F2F7`) measure ~4.3:1. Elderly-user portal is most exposed.

## Findings

### P0 — blocks usage or breaks core flow

#### P0-1. Dashboard modals are not dialogs
- Location: `app/(dashboard)/scheduler/appointment-form-modal.tsx:668`, `app/(dashboard)/scheduler/clinical-note-modal.tsx`, `app/(dashboard)/patients/patient-drawer.tsx:1180`
- Problem: A plain `<div className="fixed inset-0 z-50 ...">` is used with no `role="dialog"`, no `aria-modal`, no `aria-labelledby`, no ESC listener, no focus trap, no return-focus-on-close. `grep -rn 'role="dialog"'` across the whole `(dashboard)` tree returns zero hits. The only compliant dialog is in the portal.
- Impact: Screen-reader users can't perceive the modal; keyboard users can tab into background content; no ESC-to-close (users expect it).
- Fix:
  ```tsx
  // appointment-form-modal.tsx:668 — wrap in shadcn Dialog OR add manually:
  <div
    role="dialog"
    aria-modal="true"
    aria-labelledby="appt-modal-title"
    onKeyDown={(e) => e.key === "Escape" && onClose()}
    className="fixed inset-0 z-50 flex items-center justify-center bg-black/50"
    onClick={(e) => e.target === e.currentTarget && onClose()}
  >
  ```
  Better: migrate to `components/ui/dialog.tsx` (shadcn Radix Dialog already handles all of this).
- Effort: S (per modal)

#### P0-2. `alert()` on the patient cancel failure path
- Location: `app/portal/[slug]/mis-citas/page.tsx:282,285`
- Problem: `alert(data.error || "Error al cancelar")` and `alert("Error de conexión")`. Native `alert()` is a blocking, ugly native dialog on mobile Safari and breaks the Apple-Health design language of the portal.
- Impact: Elderly patients see a jarring system dialog mid-flow; breaks brand trust.
- Fix: Reuse the inline banner pattern already present in `/book/page.tsx:554-559`:
  ```tsx
  {cancelError && (
    <div className="flex items-center gap-2 rounded-2xl bg-red-50 px-4 py-3 text-sm text-red-700 ring-1 ring-red-200">
      <AlertCircle className="h-4 w-4 shrink-0" />
      {cancelError}
    </div>
  )}
  ```
- Effort: XS

#### P0-3. Clinical note signing uses native `confirm()` for an irreversible action
- Location: `app/(dashboard)/scheduler/clinical-note-panel.tsx:350`
- Problem: `if (!confirm("¿Firmar esta nota clínica? Una vez firmada no podrá ser editada.")) return;` — legally/medically irreversible action gated by one native dialog. No typed confirmation, no summary of what's being signed, no second chance.
- Impact: Doctor mis-clicks in the middle of a note and loses ability to edit. High regret risk.
- Fix: shadcn AlertDialog with a required checkbox ("Confirmo que he revisado los cuatro campos SOAP") and a disabled primary button until checked. Include a read-only preview of the note above the confirm button.
- Effort: M

#### P0-4. Scheduler appointment delete uses native `window.confirm()`
- Location: `app/(dashboard)/scheduler/appointment-sidebar.tsx:549`
- Problem: `if (!confirm(t("scheduler.delete_confirm"))) return;` — native browser dialog, non-branded, no "Undo" toast after delete, no warning if the appointment has an associated clinical note or payments.
- Impact: Receptionists lose data, especially in multi-office flows.
- Fix: shadcn AlertDialog + post-delete toast with `action: { label: "Deshacer", onClick: restore }`.
- Effort: S

#### P0-5. `/book` submit errors only surface at the top of the page, steps away
- Location: `app/book/[slug]/page.tsx:375-378, 553-559`
- Problem: On submit failure, `setError(body.error || "Error al crear la cita")` fires but the error banner lives above step content. On mobile, after the Confirm tap the user is scrolled at the bottom — they never see the banner.
- Impact: Public booking flow fails silently; patient assumes success and waits; no-shows rise.
- Fix: Scroll the banner into view on error, or better, render the error adjacent to the primary button:
  ```tsx
  {error && (
    <div className="rounded-lg bg-red-50 px-4 py-3 text-sm text-red-700 ring-1 ring-red-200 mt-3">
      <strong>No pudimos reservar:</strong> {error}
    </div>
  )}
  <button onClick={handleSubmit} .../>
  ```
- Effort: XS

#### P0-6. `/book` "Siguiente" button silently disables on incomplete info step
- Location: `app/book/[slug]/page.tsx:469-486, 1032-1042`
- Problem: `canGoNext()` returns false with no indication of *which* field is missing. `firstName.trim().length < 2` silently invalidates. No inline `aria-invalid`, no helper text under the offending input.
- Impact: Users (including anyone with a one-syllable first name like "Ar", "Ed") stare at a disabled button they can't explain.
- Fix: Track a `submittedStep` flag; when user taps a disabled "Siguiente", flip it and render per-field errors:
  ```tsx
  {submitted && !firstName.trim() && (
    <p className="mt-1 text-xs text-red-600">Ingresa tu nombre</p>
  )}
  <input aria-invalid={submitted && !firstName.trim()} .../>
  ```
- Effort: S

#### P0-7. Public booking has no session persistence
- Location: `app/book/[slug]/page.tsx:154-178` (state initialization)
- Problem: 5-step funnel with zero localStorage/sessionStorage. Any refresh, accidental back-tap, or network blip wipes the whole form including `selectedDate`, `selectedTime`, `firstName`, `lastName`, `phone`, `email`, `dni`, `notes`.
- Impact: Public booking is the highest-drop-off page — drop-off likely pronounced in LATAM mobile networks.
- Fix: Persist form state to `sessionStorage` on each change; rehydrate on mount.
- Effort: S

### P1 — confusing / friction / a11y fail

#### P1-1. Patient drawer X and Maximize buttons have ~24 px hit targets
- Location: `app/(dashboard)/patients/patient-drawer.tsx:385-397`
- Problem: `<button className="rounded-lg p-1 ..."><X className="h-4 w-4" /></button>` → 24 × 24 px. Apple/WCAG 2.2 target = 44 × 44 px (AA = 24 × 24 minimum for 2.2 but still fails internal spacing). On touch devices, closing the drawer is a lottery.
- Fix:
  ```tsx
  <button aria-label="Cerrar ficha del paciente" onClick={onClose}
    className="inline-flex h-10 w-10 items-center justify-center rounded-lg text-muted-foreground hover:bg-accent">
    <X className="h-4 w-4" />
  </button>
  ```
- Effort: XS

#### P1-2. Icon-only buttons without `aria-label` throughout dashboard
- Location: `app/(dashboard)/scheduler/appointment-form-modal.tsx:674-676`, `appointment-sidebar.tsx` (multiple), `patient-drawer.tsx:385, 392, 423-428`, `day-view.tsx:405-419` (unlock)
- Problem: Close X, expand, unblock, lock, unlock icons ship as bare `<button><Icon/></button>` with no accessible name.
- Fix: `aria-label="Cerrar"`, `aria-label="Desbloquear horario"`, etc. Already done well in the portal (`page.tsx:422, 451, 589, 979, 992`) — pattern exists, dashboard needs to copy it.
- Effort: XS (bulk edit)

#### P1-3. Portal tab text and details use `text-[12px]`
- Location: `app/portal/[slug]/mis-citas/page.tsx:963, 1112, 1354, 1429`
- Problem: Target audience is patients including elderly users. `text-[12px]` (12 px) is below iOS HIG minimum (13 px for body text, 11 px for captions only). At 12 px on `text-zinc-500` against `#F2F2F7` background, contrast measures ~4.3:1 (AA borderline).
- Fix: Lift body text to `text-[13px]` or `text-sm` (14 px), keep `text-[11px]` only for captions with `text-zinc-600` or darker.
  ```tsx
  // Before
  <p className="text-[12px] text-zinc-500">Ver y editar mi perfil</p>
  // After
  <p className="text-sm text-zinc-600">Ver y editar mi perfil</p>
  ```
- Effort: XS

#### P1-4. Scheduler appointment cards use `text-[11px]` + `text-[9px]`
- Location: `app/(dashboard)/scheduler/day-view.tsx:337`, `appointment-sidebar.tsx:1108, 1042, many more`
- Problem: Doctor names are rendered at 11 px; debt pills at 9 px (`text-[9px] font-bold`). Even for the receptionist power-user audience, 9 px red pills on a pastel background fail contrast at zoom-out levels commonly used on scheduler screens.
- Fix: Promote the smallest tier to 11 px minimum, 10 px only for superscript-like metadata:
  ```tsx
  // day-view.tsx:337 — bump from text-[11px] to text-xs
  <p className="text-xs truncate leading-tight" ...>
  ```
- Effort: S (design call + sweep)

#### P1-5. `text-[9px]` debt pill in day-view — contrast fails against doctor-tinted backgrounds
- Location: `app/(dashboard)/scheduler/day-view.tsx:328-331`
- Problem:
  ```tsx
  <span className="bg-red-500/15 px-1 py-0.5 text-[9px] font-bold text-red-600">
    <AlertTriangle className="h-2.5 w-2.5" />
    S/{pending.toFixed(0)}
  </span>
  ```
  On top of a `hexToPastel(doctorColor, 0.18)` card, `red-500/15` on `red-600` text measures ~3.1:1. Not AA.
- Fix: Solid fill for financial warnings.
  ```tsx
  <span className="rounded-full bg-red-600 px-1.5 py-0.5 text-[10px] font-bold text-white">
    S/ {pending.toFixed(0)}
  </span>
  ```
- Effort: XS

#### P1-6. Portal cancel confirm uses inline "No / Cancelar" micro-buttons
- Location: `app/portal/[slug]/mis-citas/page.tsx:1273-1290, 1452-1469`
- Problem: 12 px labels, 24 px height, no `aria-describedby` linking "¿Cancelar esta cita?" question to the action buttons. The destructive button is a red pill with the label "Cancelar" — ambiguous (cancel the cancel? or cancel the appointment?).
- Fix: Label destructive button "Sí, cancelar cita", safer button "Conservar cita". Raise to 40px targets.
  ```tsx
  <button className="rounded-full bg-[#FF3B30] px-4 py-2 text-sm font-semibold text-white">
    Sí, cancelar cita
  </button>
  ```
- Effort: XS

#### P1-7. Dashboard has no visible "Saving..." on the appointment form
- Location: `app/(dashboard)/scheduler/appointment-form-modal.tsx` — saving state exists (`saving` flag line ~104) but toast.success fires post-submit only
- Problem: Long form, no optimistic saved feedback *inside* the modal. If network stalls, the user reclicks "Guardar".
- Fix: Disable submit + swap label to "Guardando..." with spinner (pattern already used in portal cancel at line 1464).
- Effort: XS

#### P1-8. Portal empty state points users to online booking even when it's disabled
- Location: `app/portal/[slug]/mis-citas/page.tsx:660-668` (mobile empty state)
- Problem: The mobile `<Link href={'/book/${slug}'}>` ignores `allowOnlineBooking`; only the desktop empty state (line 772-790) branches correctly. Patients tap the CTA, land on 404/error, lose trust.
- Fix: Mirror the desktop conditional in the mobile branch:
  ```tsx
  {allowOnlineBooking ? (
    <Link href={`/book/${slug}`} ...>Agendar cita</Link>
  ) : (
    <button onClick={() => setShowContact(true)} ...>Contactar clínica</button>
  )}
  ```
- Effort: XS

#### P1-9. `/book` success screen has no "Add to calendar" or "Return to portal"
- Location: `app/book/[slug]/page.tsx:415-457`
- Problem: Patient books successfully, sees a summary, then a dead-end. No .ics, no portal link, no WhatsApp confirmation, no "Ver mis citas" if `portal_enabled`.
- Fix: Add a trio of CTAs:
  ```tsx
  <div className="flex flex-col gap-2">
    <a download="cita.ics" href={icsDataUrl} className="...">Añadir al calendario</a>
    {settings.portal_enabled && (
      <Link href={`/portal/${slug}`} className="...">Entrar a mi portal</Link>
    )}
    <a href={`https://wa.me/${org.phone.replace(/\D/g,'')}`} className="...">Contactar por WhatsApp</a>
  </div>
  ```
- Effort: M

#### P1-10. Portal profile/settings accent ring does not actually highlight on keyboard focus (missing ring-color)
- Location: `app/portal/[slug]/mis-citas/page.tsx:396-401`
- Problem: `focus-visible:ring-2 focus-visible:ring-offset-2` is set without a default `ring-color`; the dynamic `--tw-ring-color` arg works on the main button but is missing on most secondary buttons (e.g., the 40x40 profile/logout buttons at line 423, 452). Result: keyboard users see a thin transparent ring.
- Fix: Add `focus-visible:ring-zinc-900` fallback or thread `accent` consistently.
- Effort: XS

#### P1-11. Patient-drawer tag-remove X is hover-only (desktop pattern, breaks on touch)
- Location: `app/(dashboard)/patients/patient-drawer.tsx:423-428`
- Problem: `className="hidden group-hover:inline-block"` — touch devices have no hover; the X never appears. Receptionists on iPads cannot remove tags.
- Fix: Always-visible X at reduced opacity, highlight on hover/focus:
  ```tsx
  <button aria-label={`Quitar etiqueta ${tag.tag}`}
    className="opacity-60 hover:opacity-100 focus-visible:opacity-100 focus-visible:outline-primary">
    <X className="h-3 w-3" />
  </button>
  ```
- Effort: XS

#### P1-12. `/book` date picker only shows 7 days at a time; paging is unlabeled
- Location: `app/book/[slug]/page.tsx:769-788`
- Problem: ChevronLeft/Right buttons at `p-1.5` (24 px) have no `aria-label`. Screen readers announce "button button". Also no month label between pages so users don't know which week they're in.
- Fix:
  ```tsx
  <button aria-label="Semana anterior" ...><ChevronLeft/></button>
  <span className="text-xs font-medium">{monthRangeLabel}</span>
  <button aria-label="Semana siguiente" ...><ChevronRight/></button>
  ```
- Effort: XS

#### P1-13. Break-time / blocked slot label visibility depends on `:00` marks
- Location: `app/(dashboard)/scheduler/day-view.tsx:383-401`
- Problem: `{time.endsWith(":00") ? ... : <span />}` — if the block starts at `:15`, the label renders on the *next* hour row below, misrepresenting where the block begins. Clashes with the "Lock" icon positioning.
- Fix: Render the label on the first occupied slot of each block (track a `seenBlockIds` set).
- Effort: S

#### P1-14. Empty state for the doctor dashboard is unclear
- Location: `app/(dashboard)/dashboard/doctor-dashboard.tsx` (needs review but similar pattern to admin)
- Problem: When a doctor has zero upcoming appointments, the dashboard renders empty tiles. No "Crea tu primera cita" CTA, no "Link to scheduler".
- Fix: Add a zero-state hero when `todayAppts.length === 0 && upcomingCount === 0`.
- Effort: S

#### P1-15. `/book` — selected time slot relies on background color only (no border / check)
- Location: `app/book/[slug]/page.tsx:839-857`
- Problem: Selected slot uses only `backgroundColor: accentColor` (inline style) and white text — colorblind users (especially emerald/red deuteranopia) can't distinguish selected from unselected when accent is green.
- Fix: Add a redundant cue:
  ```tsx
  className={cn(
    "rounded-lg py-2.5 text-sm font-medium transition-all",
    selectedTime === slot
      ? "text-white ring-2 ring-offset-2 ring-offset-white font-bold"
      : "border border-zinc-200 bg-white text-zinc-700 hover:border-zinc-400"
  )}
  ```

### P2 — polish / consistency

#### P2-1. Three radius systems coexist
- Locations:
  - Portal: `rounded-2xl`, `rounded-3xl`, `rounded-t-3xl` (`app/portal/[slug]/mis-citas/page.tsx:626, 680, 759, 1584`)
  - Dashboard cards: `rounded-2xl` (`app/(dashboard)/dashboard/admin-dashboard.tsx:205, 226, 256, 304`)
  - Dashboard modals/inputs: `rounded-lg`, `rounded-xl` (`appointment-form-modal.tsx` — 35× `rounded-lg`, 3× `rounded-xl`)
  - Booking cards: `rounded-xl` (`app/book/[slug]/page.tsx:647, 660, 721`)
- Problem: No system codifies when to use which tier. Modal inner buttons at `rounded-lg` sit in a `rounded-xl` modal container, which sits over a portal showing `rounded-3xl` cards. Feels inconsistent across surfaces.
- Fix: Lock 3 tiers in Tailwind config:
  ```js
  // tailwind.config — radius scale
  borderRadius: {
    sm: '8px',   // inputs, chips
    md: '12px',  // buttons, small cards
    lg: '16px',  // standard cards, modals
    xl: '24px',  // hero cards, sheets
  }
  ```
  Pick one per surface type and migrate.
- Effort: M (system-wide sweep)

#### P2-2. Emerald accent is overridden by per-org `accent_color` in the portal + booking, but the sidebar icons remain hardcoded
- Location: Portal uses `accent` var everywhere (`mis-citas/page.tsx:222`); Booking uses `accentColor`; sidebar uses `text-emerald-500` hardcoded (`clinical-note-panel.tsx:410`, `day-view.tsx:456`)
- Problem: A clinic that picks `#e11d48` (rose) sees their branding in the portal and booking, then hits the scheduler where emerald leaks in the "+" icon on empty slots and the SOAP icon. Inconsistent branding.
- Fix: Reserve `text-emerald-*` for system semantic greens (success states) and use `text-primary` for accent everywhere else.
- Effort: S

#### P2-3. Hex-string colors hardcoded across patient portal
- Location: `app/portal/[slug]/mis-citas/page.tsx:192-197, 651, 762, 1249, 1282, 1395, 1416, ...`
- Problem: 30+ hard-coded iOS system hex colors (`#FF3B30`, `#34C759`, `#FF9500`, `#AF52DE`, `#8E8E93`, `#007AFF`). When Tailwind 4 supports theme-based colors, these will dark-mode differently than the rest.
- Fix: Define as CSS vars:
  ```css
  --ios-red: #FF3B30;
  --ios-green: #34C759;
  --ios-orange: #FF9500;
  --ios-purple: #AF52DE;
  --ios-blue: #007AFF;
  --ios-gray: #8E8E93;
  ```
  Use via `bg-[--ios-red]` or wrap in Tailwind plugin.
- Effort: S

#### P2-4. Buttons come in at least 6 flavors with no shared variant layer
- Location: Across dashboard
  - `rounded-lg bg-primary text-primary-foreground px-4 py-2.5` (appointment-form-modal.tsx:1323)
  - `rounded-full px-4 py-2 text-[13px] font-semibold text-white` (portal, inline style)
  - `rounded-xl border border-border/60 bg-card px-4 py-2.5` (admin-dashboard:194)
  - `rounded-lg bg-primary/10 text-primary` (scheduler template button, 722)
  - `rounded-lg px-6 py-2.5 text-sm font-semibold` (book submit, 1026)
- Problem: There's a shadcn `Button` in `components/ui/button.tsx` but it's used inconsistently. Handwritten button classes bloat the codebase and diverge.
- Fix: Enforce `Button` variant usage; add `size="lg"` for booking primary, `size="pill"` variant for portal rounded-full buttons.
- Effort: M

#### P2-5. Inputs never pass focus-color correctly
- Location: `app/(dashboard)/scheduler/appointment-form-modal.tsx:701, 709, 878, 889 (…)` and `app/book/[slug]/page.tsx:879-880`
- Problem: `focus:border-emerald-500` is hardcoded but the ring gets `--tw-ring-color: accentColor` applied inline → border jumps to emerald regardless of org accent.
- Fix: Use `focus:border-primary` (token-based) so per-org theming works.
- Effort: S (bulk replace)

#### P2-6. Portal uses `capitalize` on formatted Spanish dates that already come uppercase
- Location: `app/portal/[slug]/mis-citas/page.tsx:1109, 1209`
- Problem: `es-PE` locale formatter already emits "Lunes 23 de abril"; applying `capitalize` leaves it unchanged but on values like "—" capitalizes nothing. Harmless but noise.
- Fix: Drop `capitalize` where the formatter already handles casing. Keep it only on the day-of-week chips (`getDayOfWeek` uppercases already).
- Effort: XS

#### P2-7. `SummaryTile` onClick version uses `active:scale-[0.98]`, non-clickable version has no press-state
- Location: `app/portal/[slug]/mis-citas/page.tsx:1115-1128`
- Problem: Identical visual card; only the clickable one animates on press. No visual cue that one card is clickable and one isn't. Users tap the non-interactive ones and feel broken.
- Fix: Either make all tiles navigate somewhere useful (e.g., "Completadas" → historial filter), or add a subtle `cursor-default opacity-70` to non-interactive ones.
- Effort: XS

#### P2-8. `/book` progress bar has no step labels beyond a single line caption
- Location: `app/book/[slug]/page.tsx:533-549`
- Problem: A thin 4 px bar with text "Paso 1 de 5: Doctor" — users can't see what comes later or tap to jump back to a completed step.
- Fix: Render stepper as a list with `aria-current="step"`:
  ```tsx
  <ol className="flex gap-2 overflow-x-auto" aria-label="Progreso de reserva">
    {steps.map((s, i) => (
      <li key={s.key} aria-current={i === currentStepIdx ? "step" : undefined}
        className={cn("flex items-center gap-1.5 text-xs",
          i < currentStepIdx && "text-emerald-600")}>
        <span className="h-5 w-5 rounded-full ...">{i+1}</span>
        {s.label}
      </li>
    ))}
  </ol>
  ```
- Effort: S

#### P2-9. Doctor dashboard "Stethoscope" icon hardcoded to 16 px everywhere else uses `h-4 w-4`
- Location: widespread
- Problem: Some icons `h-3 w-3` (9 px), some `h-3.5 w-3.5`, most `h-4 w-4`, some `h-5 w-5`. No system.
- Fix: Tailwind plugin or lint rule: 3.5 for inline badges, 4 for buttons, 5 for headings. Document in `components/ui/`.

#### P2-10. Scheduler cards render debt pills + payment icons + virtual icons in `flex gap-1` — visual overflow on narrow cards
- Location: `app/(dashboard)/scheduler/day-view.tsx:303-336`
- Problem: Patient name + 3–4 conditional icons + debt pill. On 200 px office columns (common on 13" screens with 4 offices), the debt pill pushes the name to truncate at "María G…"
- Fix: Render at most 1 icon in the header row; stack badges below:

## Design system recommendations

Five tokens/patterns to codify first:

1. **Radius scale (4 tiers, documented):** `sm=8` for inputs/chips, `md=12` for buttons/small cards, `lg=16` for cards/modals, `xl=24` for hero/bottom sheets. Forbid `rounded-3xl` outside of hero-card surfaces. Document in Storybook or `components/ui/README.md`.

2. **Semantic color tokens over hex strings.** Replace portal's hardcoded `#FF3B30`/`#34C759`/`#FF9500`/`#AF52DE` with CSS variables (`--status-danger`, `--status-success`, `--status-warn`, `--status-info`). Dashboard already uses shadcn tokens — extend, don't fork.

3. **One Button source of truth.** Promote `components/ui/button.tsx` to canonical; add `pill` variant (`rounded-full`) for portal/booking and `xs` size (28 px height) to replace the many hand-rolled `text-[11px]` pill buttons. Lint/codemod hand-written `rounded-lg bg-primary ...` button patterns.

4. **Typography floor of 13 px on patient-facing surfaces, 11 px on dashboard.** No `text-[9px]` anywhere except `title`-style attribute tooltips. Add Tailwind tokens:
   ```css
   --text-caption: 11px;  /* dashboard min */
   --text-body-sm: 13px;  /* patient min */
   --text-body: 15px;
   ```

5. **Dialog primitive enforcement.** Disallow `<div className="fixed inset-0 z-50 ...">` as a modal. Every modal imports from `components/ui/dialog.tsx` (Radix), getting a11y + focus trap + ESC + click-outside for free. Add an ESLint rule or codemod that flags the anti-pattern.
